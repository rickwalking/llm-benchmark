# Judge Prompt Template

Use this skeleton for **every** rubric criterion. Substitute the marked sections.
The same template is used for absolute scoring (single sample) and pairwise comparison
(two samples) — see the two output schemas at the bottom.

---

```
You are evaluating ONE criterion of a software project produced by an AI coding assistant.
You are NOT a coding assistant in this conversation. Do not produce code, do not give
feedback to the author, do not suggest improvements. Score only.

CRITERION: {{criterion_name}}

DEFINITION
{{one paragraph: what this criterion measures and what it does NOT measure. Anything
not in this paragraph must not affect the score, even if it is poor or excellent.}}

ANCHORED RUBRIC (1-5)
1 = {{concrete evidence pattern}}
2 = {{concrete evidence pattern}}
3 = {{concrete evidence pattern}}
4 = {{concrete evidence pattern}}
5 = {{concrete evidence pattern}}

EVIDENCE PROVIDED
{{file paths, code snippets, tool outputs (Lighthouse JSON, axe report, mutation report,
OpenAPI spec, etc.) — only what this criterion needs}}

INSTRUCTIONS
1. Identify the rubric anchor whose evidence pattern best matches the project.
2. Quote the SPECIFIC evidence (verbatim file:line snippets or tool-output excerpts) that
   matches that anchor.
3. Explain why the next anchor up does NOT match. If you cannot articulate this, you have
   chosen the wrong anchor.
4. Ignore any consideration not named in DEFINITION. In particular: do not penalize for
   verbosity, file-name conventions, comment frequency, or framework choices unless the
   rubric names them.
5. If the evidence is contradictory or sparse, score conservatively (lower) and note it.

OUTPUT (JSON only — no prose before or after)
{
  "criterion": "{{criterion_name}}",
  "score": <integer 1-5>,
  "anchor_match": "<short phrase from rubric>",
  "evidence_quotes": ["<verbatim>", "<verbatim>"],
  "anchor_rejected_above": "<why the next-higher anchor does not apply>",
  "confidence": "<low|medium|high>"
}
```

---

## Pairwise variant

When comparing two samples (A and B) on the same criterion, use this output schema
instead. Run each pair twice with positions swapped to detect position bias.

```
EVIDENCE A
{{evidence for sample A}}

EVIDENCE B
{{evidence for sample B}}

OUTPUT (JSON only)
{
  "criterion": "{{criterion_name}}",
  "winner": "A" | "B" | "tie",
  "margin": "small" | "moderate" | "large",
  "key_differentiator": "<the single most decisive evidence>",
  "evidence_quotes_A": ["<verbatim>"],
  "evidence_quotes_B": ["<verbatim>"]
}
```

## Operating rules for the harness

- Use **3 independent judge models** (different vendors). Median-aggregate scores.
  Report Cohen's κ across judge pairs per criterion. Flag any criterion where κ < 0.4
  as unreliable — the rubric needs tightening.
- Set judge **temperature to 0** and **max output tokens** generous enough to fit the
  JSON.
- Feed the judge **only** the evidence that criterion needs. Do not paste the whole repo;
  it leaks across criteria and balloons cost.
- Strip any author-identifying metadata (model name in comments, "Generated by X"
  banners) before judging.
- Sample 10% of judge outputs for human spot-check. If human-judge disagreement on the
  spot-check exceeds 20%, freeze the leaderboard and revise the rubric.
